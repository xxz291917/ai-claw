# 技术问答

先搜索，后回答。确保回答有据可查，避免臆测。

## 触发条件

以下情况应使用此流程：
- 用户询问技术方案 / 架构设计 / 最佳实践
- 用户询问业务逻辑 / 功能实现细节
- 用户问 "怎么实现 XX" / "XX 的原理是什么" / "为什么用 XX 而不是 YY"
- 用户询问 API 用法、框架对比、技术选型
- 用户对现有系统的实现细节有疑问

## 流程

### 1. 分析问题

从用户消息中提取：
- **主题**：问的是什么领域（前端/后端/数据库/DevOps/业务...）
- **具体问题**：核心诉求是什么
- **上下文**：是否涉及特定项目或通用知识

判断需要哪些信息源（按优先级）：

| 信息源 | 适用场景 | 工具 |
|--------|---------|------|
| 项目知识库 | 涉及团队业务、已有决策、需求文档 | `notion-rag__search`（MCP，如可用） |
| 项目代码 | 涉及现有实现、代码架构 | `code-rag__search_code`（MCP，如可用） |
| 网络搜索 | 通用技术问题、最新动态、最佳实践 | `web_search` |
| 技术文档 | 需要阅读官方文档、API 参考 | `web_fetch` |

### 2. 搜索上下文

**至少使用 2 个信息源**，除非问题非常明确且你有十足把握。

#### 搜索策略

- 使用多个关键词组合搜索，不要只搜一次
- 中文问题也尝试用英文关键词搜索（技术术语通常英文更准确）
- 如果第一轮搜索结果不够，调整关键词再搜一轮

#### 项目内搜索（有相关工具时）

```
notion-rag__search: "<功能名/模块名/业务关键词>"
code-rag__search_code: "<函数名/类名/技术关键词>"
```

#### 网络搜索

```
web_search: "<技术问题关键词> best practices 2025"
web_search: "<框架名> <具体功能> example"
```

对于有价值的搜索结果，用 `web_fetch` 获取详细内容：
```
web_fetch: "<url>"
```

### 3. 整合回答

将搜索到的信息整合成结构化回答：

```
## 回答

<简明扼要的直接回答，1-3 句话>

## 详细说明

<展开解释，包含：>
- 核心概念和原理
- 具体实现方式或步骤
- 代码示例（如适用）

## 对比/权衡（如涉及方案选择）

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| A    | ...  | ...  | ...     |
| B    | ...  | ...  | ...     |

**推荐**：<推荐方案及理由>

## 参考来源

- <来源 1>：<简述该来源提供了什么信息>
- <来源 2>：<简述>
```

## 回答原则

1. **先搜后答**：不要凭记忆直接回答，先搜索确认
2. **标注来源**：回答中引用的信息要标注出处
3. **区分事实和建议**：已验证的事实 vs 个人建议要区分开
4. **承认不确定**：如果搜索后仍不确定，明确说明而不是猜测
5. **实用优先**：给出可直接使用的代码、命令、配置，而非纯理论
6. **项目优先**：如果问题涉及团队项目，优先参考项目知识库和代码，而非通用答案

## Notes

- 对于简单事实性问题（如 "Node.js LTS 版本是多少"），一次搜索即可
- 对于复杂方案类问题，至少搜索 2-3 轮，覆盖不同角度
- 如果用户追问细节，可以针对性地深入搜索
- 搜索结果过长时，提取关键段落，不要全文搬运
