# AI Hub 交互流程设计

> AI 工程助手 — 把工程团队中大量重复、耗时的低价值工作自动化
>
> 原则：AI 负责执行与建议，人类负责最终决策与批准

---

## 1. 项目定位

建设一个"AI 工程助手（AI Hub）"，覆盖三个核心场景：

| 场景 | 触发方式 | 交互模式 | 优先级 |
|------|---------|---------|--------|
| 故障自愈 | Sentry webhook（自动） | 流水线 | Phase 1 |
| 需求交付 | Notion 状态变更（自动） | 流水线 + 迭代 | Phase 2 |
| 自然语言 DevOps | 飞书 @AI（手动） | 对话式 | Phase 2 |

---

## 2. 交互架构：方案 C（流水线 + 对话融合）

自动触发的场景（Sentry、Notion）走**流水线模式**，人工触发的场景走**对话模式**。共用同一个 AI 引擎。

### 2.1 系统总体架构

```
┌──────────────────────────────────────────────────────────┐
│                      AI Hub 服务                          │
│                                                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │ Webhook 网关 │  │  任务调度器   │  │  状态机引擎  │      │
│  │             │  │             │  │             │      │
│  │ - Sentry    │→│ - 去重判断   │→│ - 流水线模式 │      │
│  │ - Notion    │  │ - 优先级    │  │ - 对话模式   │      │
│  │ - GitHub    │  │ - 并发控制   │  │ - 阶段推进   │      │
│  └─────────────┘  └─────────────┘  └──────┬──────┘      │
│                                           │              │
│                    ┌──────────────────────┤              │
│                    ▼                      ▼              │
│  ┌──────────────────────┐  ┌──────────────────────┐     │
│  │   Claude Code SDK    │  │     飞书 Bot SDK      │     │
│  │                      │  │                      │     │
│  │ - 代码分析/修改       │  │ - 卡片消息通知        │     │
│  │ - 测试执行            │  │ - 按钮交互（审批）     │     │
│  │ - PR 创建             │  │ - 对话式交互          │     │
│  └──────────────────────┘  └──────────────────────┘     │
│                    │                      │              │
│                    ▼                      ▼              │
│  ┌──────────────────────────────────────────────┐       │
│  │           任务存储 (SQLite / .jsonl)           │       │
│  │  - 任务状态、阶段、审批记录、执行日志           │       │
│  └──────────────────────────────────────────────┘       │
└──────────────────────────────────────────────────────────┘
         │                │               │
         ▼                ▼               ▼
   ┌──────────┐    ┌──────────┐    ┌──────────┐
   │  Sentry  │    │  GitHub  │    │  Notion  │
   └──────────┘    └──────────┘    └──────────┘
```

### 2.2 核心组件

| 组件 | 职责 |
|------|------|
| Webhook 网关 | 接收外部事件（Sentry 报警、Notion 变更、GitHub PR 事件） |
| 任务调度器 | 去重、优先级排序、并发控制（避免 AI 同时改同一个仓库） |
| 状态机引擎 | 管理每个任务的生命周期和阶段推进，流水线和对话两种模式共用 |
| Claude Code SDK | AI 执行引擎，负责分析、写代码、跑测试、提 PR |
| 飞书 Bot | 人机交互层 — 通知 + 审批 + 对话 + 提问 |

### 2.3 外部系统角色

```
Sentry  = 触发源（故障报警）
GitHub  = 代码主战场（分支、PR、CI、合并）
Notion  = 三重角色:
          1. 触发源 — 需求文档更新触发 AI
          2. 工作台 — AI 把方案写回 Notion
          3. 审批台 — 人在 Notion 里批注/直接修改，AI 迭代
飞书     = 两重角色:
          1. 通知通道 — 卡片消息 + 按钮审批
          2. 对话通道 — AI 提问、工程师指令、DevOps 交互
```

## 3. 场景 1：故障自愈流程

### 3.1 流程概览

```
Sentry webhook
    │
    ▼
[阶段1] 接收评估（自动）
  - 去重检查（同一 issue 不重复处理）
  - 严重性评估（P1-P4）
  - 分配优先级
    │
    ▼
[阶段2] AI 分析（自动，无需审批）
  Claude Code:
  - 读取 Sentry 堆栈信息
  - 定位相关代码文件
  - 分析根本原因
  - 查询历史类似案例
    │
    ▼
[飞书通知] 诊断报告卡片
  ┌────────────────────────┐
  │ 🔴 P1 故障告警           │
  │                        │
  │ 错误: TypeError         │
  │ 文件: handler.ts:42     │
  │ 根因: 未处理 null       │
  │ 置信度: 92%             │
  │ 影响: 1.2k 用户         │
  │                        │
  │ [🔧 生成修复]           │
  │ [👀 仅查看详情]         │
  │ [🚫 忽略]              │
  └────────────────────────┘
    │
    ▼ 工程师点击 [生成修复]
    │
[阶段3] AI 修复（审批后执行）
  Claude Code:
  - 编写修复代码
  - 生成回归测试
  - 运行测试
  - 创建 Git 分支 + PR
    │
    ▼ CI 自动运行
    │
[飞书通知] PR 就绪卡片
  ┌────────────────────────┐
  │ ✅ 修复 PR 已就绪       │
  │                        │
  │ PR #89                 │
  │ 改动: 3文件 +12行       │
  │ 测试: 8通过 0失败       │
  │ CI: ✅ 通过             │
  │                        │
  │ [✅ 合并 PR]            │
  │ [📝 查看代码]           │
  │ [🔄 要求修改]           │
  └────────────────────────┘
    │
    ▼ 工程师点击 [合并 PR]
    │
[阶段4] 合并部署
  - Merge PR
  - GitHub Actions 自动部署
  - 飞书通知"故障已修复并部署"
```

### 3.2 关键设计点

- **两次人工门禁**："是否生成修复"和"是否合并 PR"，都在飞书卡片上一键操作
- **AI 分析阶段全自动**：不打扰工程师，只在有结果时推送
- **置信度指标**：让工程师快速判断是否值得 AI 修复
- **"要求修改"按钮**：工程师可以打回，AI 根据反馈重新修改

## 4. 场景 2：需求到交付流程

### 4.1 流程概览

```
Notion 需求文档状态变更为"待开发"
    │
    ▼
[阶段1] 需求解析（自动）
  Claude Code:
  - 读取 Notion 需求文档
  - 理解业务意图
  - 识别技术约束
    │
    ▼
  AI 生成实施方案 → 写入 Notion 子页面
  - 架构设计
  - 技术选型
  - 任务拆解（含子任务列表）
  - 工时预估
    │
    ▼
[飞书通知] "实施方案已就绪，请在 Notion 查看"
    │
    ▼
[阶段2] 人机协作迭代（可多轮）
  ┌─────────────────────────────────────────────┐
  │                                             │
  │  工程师反馈方式：                              │
  │  A. 在 Notion 批注："数据库改用 PostgreSQL"    │
  │  B. 直接编辑 Notion 内容（更常见）             │
  │                                             │
  │         ▼                                   │
  │                                             │
  │  AI 检测变更（diff + 读批注）                  │
  │  - 保留人工修改内容（不覆盖）                  │
  │  - 补全受影响的关联部分                        │
  │  - 润色和优化表述                              │
  │  - 对潜在问题给出建议（而非直接改）             │
  │  - 遇到歧义时通过飞书提问                      │
  │                                             │
  │         ▼                                   │
  │                                             │
  │  更新方案 → 飞书通知"方案已更新"              │
  │                                             │
  │  循环，直到工程师在 Notion 标记"方案通过" ✅   │
  └─────────────────────────────────────────────┘
    │
    ▼ 方案通过
    │
[阶段3] 代码生成
  Claude Code（按任务拆解逐个执行）:
  - 创建 feature 分支
  - 逐个子任务生成代码（每个独立 commit）
  - 为每个子任务生成单元测试
  - 运行单元测试
    │
    ▼
[阶段4] E2E 测试
  Claude Code:
  - 运行端到端测试
  - 生成测试报告（覆盖率、通过/失败、性能数据）
  - 测试报告写回 Notion 子页面
  - 创建 PR
    │
    ▼
[飞书通知] PR 就绪卡片
  ┌────────────────────────┐
  │ ✅ 代码已就绪           │
  │                        │
  │ PR #102                │
  │ 5个子任务全完成         │
  │ 测试覆盖: 87%          │
  │ E2E: 12/12 通过        │
  │                        │
  │ [✅ 合并]              │
  │ [📝 查看代码]          │
  │ [🔄 要求修改]          │
  └────────────────────────┘
    │
    ▼ 工程师审批
```

### 4.2 Notion 迭代协作原则

**人工修改 > AI 修改**：AI 永远不覆盖工程师直接编辑的内容。

AI 对人工修改的处理策略：

| 策略 | 说明 | 示例 |
|------|------|------|
| 保留 | 人工修改的内容不覆盖 | 工程师改了数据库选型，AI 不回退 |
| 补全 | 补全受影响的关联部分 | 数据库改了，AI 自动更新 ORM 和连接池配置 |
| 润色 | 优化表述、统一格式、补充细节 | 统一术语、补充缺失的技术细节 |
| 建议 | 对潜在问题给出建议（不直接改） | "建议连接池从 mysql2 换成 pg-pool" |
| 提问 | 遇到歧义或需要决策时通过飞书提问 | "你改用了 PostgreSQL，JSON 字段用 jsonb 还是 text？" |

### 4.3 与场景 1 的关键差异

- **Notion 双向集成**：AI 写方案到 Notion，人在 Notion 批注/直接修改，AI 读取变更迭代
- **多轮迭代环**：方案阶段可反复修改直到人工标记"通过"
- **分步代码生成**：按任务拆解逐个生成，每个子任务独立 commit
- **测试报告写回 Notion**：形成需求 → 方案 → 代码 → 测试的完整闭环文档

## 5. 场景 3：自然语言 DevOps

### 5.1 交互模式：对话式

工程师在飞书群 `#devops-ai` 中直接 @AI 交互。

```
👤 工程师:
@AI 帮我把 user-service 部署到 staging

🤖 AI:
收到。当前 user-service 状态：
- main 分支最新 commit: abc123 "feat: add avatar upload"
- staging 当前版本: def456 (落后 3 个 commit)
- CI 状态: ✅ 通过

⚠️ 发现一个注意项：
commit abc123 新增了 S3_BUCKET 环境变量，staging 尚未配置

请确认：
[🚀 继续部署，我稍后配环境变量]
[⏸️ 先暂停，我去配好再部署]
[📋 帮我一起配置环境变量]
```

### 5.2 设计要点

| 要点 | 说明 |
|------|------|
| 主动检测阻塞项 | 部署前自动检查环境变量、数据库迁移、依赖变更等 |
| 实时进度反馈 | 飞书卡片动态更新部署进度（不用反复刷 CI 页面） |
| 部署后持续监控 | 部署完成后 AI 自动关注错误率，异常时主动通知 |
| 一键回滚 | 出问题时飞书卡片直接回滚，不用登录服务器 |

### 5.3 典型交互场景

```
场景 A: 部署
  "@AI 部署 user-service 到 staging"
  → 检查状态 → 发现阻塞项 → 协助解决 → 部署 → 监控

场景 B: 测试
  "@AI 跑一下 payment-service 的回归测试"
  → 执行测试 → 实时反馈进度 → 返回结果 → 失败时分析原因

场景 C: 查询
  "@AI staging 环境现在什么状态"
  → 汇总各服务版本、健康状态、最近部署记录

场景 D: 回滚
  "@AI 把 user-service 回滚到上一版本"
  → 确认回滚目标 → 执行 → 验证健康检查
```

## 6. 能力架构：Tool / Skill / 内置能力

### 6.1 三层能力模型

```
┌───────────────────────────────────────────────────────┐
│  编排层（系统代码实现，非 Tool/Skill）                   │
│  Webhook 网关 / 任务调度 / 状态机 / 审批流               │
└───────────────┬───────────────────────────────────────┘
                │
  ┌─────────────┼─────────────────┐
  ▼             ▼                 ▼
┌──────────┐ ┌────────────┐ ┌──────────────┐
│ Skill 层 │ │ Tool 层    │ │ 内置能力层    │
│ (流程知识)│ │ (外部对接) │ │ (Claude Code) │
│ 告诉 AI  │ │ 让 AI 能   │ │ AI 天生就    │
│ "怎么做" │ │ "连接外部" │ │ "会做"       │
└──────────┘ └────────────┘ └──────────────┘
```

### 6.2 内置能力（Claude Code 自带，无需开发）

| 能力 | 内置工具 | 用途 |
|------|---------|------|
| 读代码 | read | 所有场景 |
| 改代码 | write / edit | 故障修复、需求开发 |
| 跑命令 | bash | 测试(`npm test`)、Git(`git`, `gh`)、部署脚本 |
| 搜代码 | grep / glob | 定位相关文件 |
| Git/PR | bash (gh CLI) | 创建分支、提交、创建 PR、合并 |

### 6.3 自定义 Tool（需开发，约 6 个，每个 50-100 行）

**感知类（读取外部信息）：**

| Tool | 输入 | 输出 | 用途 |
|------|------|------|------|
| sentry_query | issue_id | 错误详情、堆栈、影响用户数 | 故障自愈 |
| notion_read | page_id | 页面内容、批注、子页面 | 需求交付 |
| notion_diff | page_id, since | 变更内容、新批注 | 需求迭代 |

**执行类（写入外部系统）：**

| Tool | 输入 | 输出 | 用途 |
|------|------|------|------|
| notion_write | page_id, content | 成功/失败 | 写方案/报告 |
| lark_notify | 卡片类型, 数据, 目标 | message_id | 所有场景通知 |
| deploy_action | service, env, action | 部署状态 | DevOps |

### 6.4 Skill（Markdown 流程指南，约 4 个）

```
skills/
├── fault-healing.md        ← 故障自愈流程
│     分析阶段: sentry_query → read 代码 → 评估严重性 → lark_notify
│     修复阶段: 创建分支 → 修代码 → 加测试 → 跑测试 → gh pr create
│     决策规则: 置信度<60%只给报告 / 改动>50行标记复杂 / 涉及DB禁止
│
├── requirement-delivery.md ← 需求交付流程
│     解析: notion_read → 生成方案 → notion_write
│     迭代: notion_diff → 保留人工修改 → 补全 → 润色 → 建议 → 提问
│     编码: 逐任务 commit → 单元测试 → E2E → 报告写回 Notion
│
├── devops-assistant.md     ← DevOps 操作指南
│     部署前检查 / 部署执行 / 部署后监控 / 回滚流程
│
└── code-standards.md       ← 团队编码规范（所有场景引用）
```

### 6.5 两层编排：不要混淆

```
外层编排（你写的代码 ~300 行）：
  任务级别的状态推进 + 人工审批
  [分析] ──人审批──► [修复] ──人审批──► [合并]
  就是 if/else 和数据库状态更新，不涉及 AI 智能

内层编排（Claude Code SDK 自带，不用写）：
  单步骤内 AI 自主迭代调用工具
  LLM: "我需要看堆栈" → sentry_query
  LLM: "定位到 handler.ts" → read
  LLM: "看看调用方" → read (另一个文件)
  LLM: "根因找到了" → 输出分析报告
```

**你的代码只管"什么时候让 AI 行动、行动完了通知谁"。**
**Claude Code SDK 管"AI 怎么思考和行动"。**

## 7. 多模型策略

### 7.1 当前策略：先用 Claude Code SDK 跑通全流程

Phase 1 不引入其他模型，所有 AI 任务都通过 Claude Code SDK 执行。

### 7.2 后续扩展：按需引入其他模型降本

| 任务类型 | 当前（Phase 1） | 后续优化 |
|---------|----------------|---------|
| 代码修复/生成 | Claude (Code SDK) | 保持 Claude |
| 报警初筛/分级 | Claude (Code SDK) | → Kimi/DeepSeek（便宜） |
| 需求文档解析 | Claude (Code SDK) | 保持 Claude |
| 方案润色/优化 | Claude (Code SDK) | → Kimi/DeepSeek（中文好） |
| 通知文案生成 | Claude (Code SDK) | → 最便宜的模型 |

扩展时：简单任务直接调 API，需要 Tool Calling 的任务自建轻量 Agentic Loop（~60 行）。

### 7.3 统一接口设计（预留）

```typescript
// 预留统一接口，后续扩展时不需要改外层编排代码
async function runAgent(params: {
  task: string;
  tools?: Tool[];
  model?: "claude" | "kimi" | "deepseek";
}): Promise<AgentResult> {
  if (!params.model || params.model === "claude") {
    return runClaudeCodeSDK(params);
  }
  return runGenericAgenticLoop(params);  // 后续实现
}
```

## 8. 技术选型

### 8.1 技术栈

| 组件 | 选择 | 理由 |
|------|------|------|
| 运行时 | Node.js 22+ (TypeScript ESM) | 与 OpenClaw 一致 |
| Web 框架 | Hono | 轻量、TypeScript 原生、OpenClaw 在用 |
| AI 引擎 | @anthropic-ai/claude-code (SDK) | 自带 read/write/bash 工具，省去 80% 工作量 |
| 数据库 | SQLite | 参考 OpenClaw，单实例够用，后期可换 PostgreSQL |
| 任务队列 | 进程内队列 | 参考 OpenClaw 的 command-queue.ts |
| 会话存储 | .jsonl 文件 | 与 OpenClaw 一致 |
| 飞书集成 | @larksuiteoapi/node-sdk | 官方 SDK |
| Notion 集成 | @notionhq/client | 官方 SDK |
| GitHub | gh CLI（Claude Code 内部调用） | 最简单可靠 |
| 部署 | Docker + 现有基础设施 | — |

### 8.2 任务隔离方式

```
故障修复 → 独立 git worktree（避免并发冲突）
需求开发 → 独立 feature branch
DevOps   → 无代码修改，直接执行命令
```

## 9. 错误处理与安全边界

### 9.1 错误处理分级

| 错误类型 | 处理方式 |
|---------|---------|
| AI 分析失败 | 自动重试 2 次 → 失败则通知工程师"请人工处理" → 降级为传统流程 |
| AI 修复失败 | 测试不通过 → 自动回退代码 → 通知工程师 → 保留分支供参考 |
| 部署失败 | 健康检查失败 → 自动回滚 → 通知工程师 |
| 飞书/Notion 不可用 | 降级为日志记录 → 任务继续执行 → Web Dashboard 可查看 |
| Claude API 限流 | 超时/限流 → 排队等待重试 → 多 API Key 轮转（参考 OpenClaw Auth Profile） |

**核心原则：AI 失败时降级为人工流程，永远不阻塞团队。**

### 7.2 安全红线（不可逾越）

1. AI 永远不能直接部署到生产环境（必须人工确认）
2. AI 永远不能删除分支、force push、或修改 main 分支保护规则
3. AI 永远不能访问生产数据库（只能读日志和监控数据）
4. AI 生成的代码必须通过 CI（lint + test）才能进入合并流程
5. 所有 AI 操作记录到审计日志（谁触发、AI 做了什么、结果如何）
6. 单次任务 Token 消耗上限（防止 AI 死循环烧钱）

## 10. 自主权分级

| 操作类型 | 自主权级别 | 说明 |
|---------|-----------|------|
| 分析、诊断、生成报告 | 全自动 | 低风险，无副作用 |
| 读取代码和配置 | 全自动 | 只读操作 |
| 写入 Notion 方案 | 全自动 | 可被人工覆盖 |
| 创建分支和修改代码 | 自动执行 | 隔离在分支中，不影响 main |
| 创建 PR | 自动执行 | PR 本身需要人工审批才能合并 |
| 合并 PR | 需人工审批 | 飞书卡片一键操作 |
| 部署到 staging | 需人工确认 | 飞书确认后执行 |
| 部署到生产 | 必须人工确认 | 不可跳过 |
| 回滚 | 需人工确认 | 紧急情况飞书一键回滚 |
